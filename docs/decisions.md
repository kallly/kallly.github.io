# Technical Decisions

Append-only log of non-obvious choices and why. One entry per decision; newest at top. Skip anything already fully explained by a doc elsewhere ŌĆö link instead of duplicating.

## Wave Analysis: `carry` gated by `timeDeficit`, `timeDeficit` made cumulative ŌĆö fixed a runaway clear-time underestimate
`analysisService.evaluateWaveDamage`'s `carry` (overkill PV passed to the next enemy) used to propagate unconditionally whenever a kill overshot an enemy's HP, sized as `damage - health` where `damage` includes the enemy's full path-crossing exposure budget (`timeToCross * rate`, cf. `computeEnemyDamage`). For a weak enemy killed almost instantly by a strong loadout, that budget vastly exceeds its actual HP, so nearly the whole budget became "free" `carry` ŌĆö and this compounded across every enemy in the wave, letting a long run of trash mobs manufacture enough phantom carry to one-shot a much later high-HP enemy with zero real damage dealt to it. Reported symptom: a Wave 45 boss with 1,200,000 HP showed the *entire* wave clearing in 44s at only 10,000 DPS ŌĆö mathematically impossible even if that DPS had focus-fired the boss alone for the whole 44s (440,000 damage max). **Fix, confirmed with the user:** `carry` is now kept for the next enemy only if `timeDeficit` is still `> 0` right after the kill ŌĆö i.e. only if a next enemy is already queued up because this kill overran `ENEMY_SPAWN_GAP_SECONDS`; otherwise the leftover is discarded (`carry = 0`), since there's no real target yet to receive it. This required making `timeDeficit` itself cumulative (`timeDeficit = max(0, timeDeficit + killTime - ENEMY_SPAWN_GAP_SECONDS)`, floored at `0`, instead of being recomputed fresh from only the last kill) so a real backlog can build across several slow kills and drain back down gradually across several fast ones. **Why:** ties `carry`'s legitimacy to the same backlog signal that already governs exposure-window shrinkage, instead of an unbounded per-enemy budget with no relationship to real elapsed time. **Applies to:** don't reintroduce unconditional `carry` propagation, and don't let `timeDeficit`'s floor go negative (a negative deficit, subtracted in `timeToCross`, would let an enemy's exposure window exceed its own full path-crossing time). See [features/wave-analysis.md](features/wave-analysis.md) steps 5/5b.

## Wave Analysis `timeDeficit`: fixed 1s spawn gap, not reset on survival/unavailable
`analysisService.evaluateWaveDamage` models enemies spawning `ENEMY_SPAWN_GAP_SECONDS` (`1`, fixed constant) apart: when a kill's `killTime` exceeds that gap, the excess is subtracted from the next enemy's `timeToCross` via `timeDeficit` (see [features/wave-analysis.md](features/wave-analysis.md)). Two choices were confirmed with the user rather than derived: (1) the 1s gap is a flat constant, not read from any per-enemy/per-wave data ŌĆö no such field exists in `hardcore_data.json`; (2) unlike `carry`, `timeDeficit` is **not** reset to 0 when an enemy survives or is `unavailable` ŌĆö it only changes on an actual kill (based on that kill's own `killTime`), otherwise it passes through unchanged to whichever enemy is next, including across group boundaries. **Why:** the user's mental model is "the wave clock keeps advancing regardless of whether the tower actually managed to kill the current enemy" ŌĆö a survivor doesn't buy the next enemy a fresh window, so the debt outlives it. **Applies to:** don't "fix" this into resetting on survival by analogy with `carry` ŌĆö that's a different, deliberately asymmetric mechanic. If the 1s constant is ever questioned, re-confirm with the user before changing it or making it data-driven.

## Wave Analysis risk badge: 20% margin, HIGH forced on any survivor
`analysisService.computeWaveRisk()` classifies a wave's clear time against its Wave Timer as LOW (`clearTime Ōēż 80%` of the timer), MEDIUM (between 80% and 100%), or HIGH (over 100%) ŌĆö but if the wave isn't fully cleared (`allKilled === false`), the badge is forced to HIGH regardless of `clearTime`, which is meaningless in that case (it only accumulates for enemies that were actually killed). **Why:** both the 20% buffer and the forced-HIGH-on-defeat behavior were explicit choices confirmed with the user rather than derived from any in-game rule ŌĆö there's no official "safety margin" number, and a wave with survivors is worse than merely "cutting it close" so it must never render as anything but the most severe badge. **Applies to:** don't change the margin or the defeat-overrides-risk behavior without re-confirming with the user; any future addition of intermediate risk tiers should keep "not fully cleared ŌćÆ HIGH" as an unconditional override, checked first.

## `WAVE_TIMER_SECONDS` is duplicated from `pages/hardcore_damage_forecast.js`, with an intentional divergence on waves 43/45
The official per-wave Hardcore "Wave Timer" (seconds before the next wave launches) isn't in any `data/*.json` file ŌĆö it's hardcoded as `WAVE_TIMER_RANGES` in `pages/hardcore_damage_forecast.js`, a standalone script outside the `src/` bundle (plain IIFE, no ES modules ŌĆö see [features/hardcore-pages.md](features/hardcore-pages.md)) that can't be imported from `analysisService.js`. The table is duplicated there under the same name, **except** waves 43 and 45: the public page caps them at an assumed `300` (needed to turn their HP into a displayable DPS number on a chart), while `analysisService`'s copy uses literal `Infinity`. **Why:** the two consumers need different semantics for "no official timer" ŌĆö the forecast page needs *some* finite number to compute a ratio; the risk badge needs "there is no next wave to be late for," i.e. always safe regardless of clear time. Reusing the 300s cap here would make a wave 43/45 clear that takes, say, 350s incorrectly render as HIGH risk. **Applies to:** if the real Wave Timer values ever change (e.g. a Roblox game update), update both copies ŌĆö there's no single source of truth to fix once. Don't try to unify the two into a shared constant without checking with the user first (same caution as unifying the pages in general, per [features/hardcore-pages.md](features/hardcore-pages.md)).

## `detections` level parsing only counts a parenthesized qualifier if it says "Tower"
`tds_stats.json`'s `detections.{hidden,lead,flying}` field is free text, not a clean number: `"Level 2+"`, `"Level 0"` (no `+`), `"Level 4B+"` (branch tier), `"Level 2+ (Splash Damage only)"`, `"Level 2+ (Tower & Tommy Goons)"`, or even multiple `"Level N (qualifier)"` groups concatenated with no separator (Commander/Kingpin multi-path troops). `analysisService.parseDetectionLevel()` extracts the leading number as the "N and above" threshold, but when a group has a parenthesized qualifier, it's only used if that qualifier's text contains "Tower" (case-insensitive) ŌĆö otherwise the detection is treated as `null` (never unlocked), even though a number is present in the string. **Why:** the game data records detection at a finer grain than this app models ŌĆö qualifiers like `"Splash Damage only"`, `"Rifleman & Riot Guard only"`, `"Converted Hologram Only"`, or path names (`"Top Path"`/`"Bottom Path"`) refer to a specific attack mode or a summoned unit, not the tower's own base DPS number that `evaluateWaveDamage` actually uses ŌĆö counting them would overstate what the modeled `tower.dps` can hit. Confirmed with the user as the simplest rule that's still meaningfully correct, rather than hand-parsing every qualifier's game semantics. **Applies to:** any future code reading `detections` must go through `parseDetectionLevel()` (see [features/wave-analysis.md](features/wave-analysis.md)) rather than re-parsing the string ŌĆö it stays the single source of truth for this heuristic, including its known limitation that multi-path troops whose qualifiers never say "Tower" (e.g. `"(Top Path)"`) resolve to no detection at all.

## Wave Analysis uses a static per-tower formula, not a time-based simulation
Two more sophisticated approaches were built and discarded before landing on `analysisService.evaluateWaveDamage` (see [features/wave-analysis.md](features/wave-analysis.md)): (1) a discrete-event simulation tracking each enemy's exact position along the path over time, with towers switching target on overtakes; (2) a single "effective DPS" blended from a coverage profile merged across all towers. **Why:** both were rejected as too complex/unreliable in practice (the first especially) for what's meant to be a rough admin planning aid, not a combat replay. **Applies to:** don't reintroduce per-enemy timeline tracking here without being asked ŌĆö the per-tower `pathCoverage` fraction ├Ś per-enemy `duration/speed` formula, cascaded with overkill carried to the next enemy in wave order, is the deliberately-chosen simpler replacement. If more fidelity is ever wanted, prefer extending this formula incrementally over reviving the old simulation.

## Path tracing (and only path tracing) stays out of saveService
Zones (`PolygonModel`) and labels (`TextLabelModel`) are included in save/share/autosave; `PathModel` deliberately is not. **Why:** enemy paths are an admin analysis aid tied to the current session, not something a normal player's save should carry. **Applies to:** any future admin-only feature ŌĆö default to isolating it like `PathModel` unless there's a specific reason players should get it in their saves (zones/labels earned inclusion because they're map-annotation tools useful to any user, not just admins reasoning about paths).

## Firebase auth kicked off without await
`initCollab()` is called but not awaited early in `app.js`, before map/troop data loads. **Why:** keeps Firebase's auth round-trip off the critical rendering path; the app can start rendering before collab is ready. It's only awaited immediately before `canvasRenderer.start()`. **Applies to:** don't refactor this to `await initCollab()` at the top of startup ŌĆö that would reintroduce a network-bound delay before first paint.

## Two save/share formats instead of one
`createSaveData` (verbose) vs `createCompactSaveData` (dictionary-encoded). **Why:** verbose is human-readable (Save/Load textarea, localStorage) but too large for a URL once LZString-compressed for bigger layouts; compact trades readability for size. **Applies to:** `parseSaveData` must keep auto-detecting shape (`version: 2` + a `t` array ŌćÆ compact) so callers never need to know which format they got ŌĆö don't add a code path that assumes one format.

## `admin.html` is a hand-synced clone, not a template or a second bundle
**Why:** the project has no templating system, and one shared `dist/app.min.js` powers both pages via a body class (`admin-mode`) rather than separate builds ŌĆö simpler than introducing build-time templating for one extra page. **Applies to:** accept the manual-sync cost (see [features/admin-mode.md](features/admin-mode.md)) rather than proposing a templating engine for this alone.

## Map key `"Wrecked Battlefiel"` keeps its typo
**Why:** it's used verbatim as a dictionary key across `data/maps.json`, the map `<select>`, `?map=` query params in existing published guide-page links, and saved layouts. Fixing the typo would silently break every existing external link/save. **Applies to:** never "fix" this key; the *displayed* name is already spelled correctly elsewhere (image filenames, page titles) ŌĆö only the JSON key is wrong, permanently.
