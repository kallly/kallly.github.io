# Wave Analysis (admin)

Admin-only panel (`#analysisPanel`, opened via the `openAnalysis` toolbar button) that estimates whether currently placed towers can handle a chosen Hardcore wave.

## Data
Lazily loaded on first panel open via `analysisService.loadAnalysisData()` — fetches `data/tds_stats.json` (combat stats) and `data/hardcore_data.json` (wave/enemy roster) in parallel, caches the result so re-opening the panel doesn't re-fetch. See [../data/hardcore-stats.md](../data/hardcore-stats.md).

## Algorithm (`analysisService.evaluateWaveDamage`)
A deliberately simple static formula, not a time-based simulation — two earlier attempts (a discrete-event simulation tracking exact enemy position over time, then a single blended "effective DPS" derived from a merged coverage profile) both proved too complex/unreliable in practice. See [../decisions.md](../decisions.md) for why this replaced them.

1. The wave's `{count, enemy, modifiers}` groups are expanded into individual enemy instances, in `wave.enemies` order (groups preserved, `count` copies each) — this order is what the damage cascade (step 5) follows.
2. Each placed tower gets a `pathCoverage` fraction: `mapModel.getPathLengthInCircle(troop.x, troop.y, troop.range) / mapModel.getTotalPathLength()` — the same per-tower coverage math already used for the Placement panel's "Path in Range" readout, not a new calculation.
3. For each enemy instance: `damage = Σ_towers (timeToCross) * tower.pathCoverage * tower.dps`, where `timeToCross = max(0, duration / enemy.speed - timeDeficit)` — `duration` is `mapModel.getPathDuration()` (seconds to cross the whole path at speed 1), enemy speed comes from `hardcore_data.json`'s free-text `Speed` field via `analysisService.parseEnemySpeed()` (flagged `approximate: true` when the raw value wasn't already a clean number — surface this in the UI, don't present it as exact). `timeDeficit` is the spawn-timing carry-over described in step 5b below. Towers are summed individually per enemy — two towers covering *different* portions of the path don't get treated as if either alone covered the union of both (that was the bug in the abandoned "effective DPS" attempt).
4. **Targeting restrictions**: before summing, `computeEnemyDamage` drops any tower that can't hit this enemy (`analysisService.canTowerHitEnemy()`). An enemy is Hidden/Flying/Lead if `hardcore_data.json`'s `Hidden`/`Flying`/`Lead` field parses truthy (`parseEnemyFlag()`) or the wave group's `modifiers` names the flag (needed for enemies like Cursed Skeleton whose base field is `"Variable"`). A tower can hit that enemy only if its placed `level` meets the threshold in `tds_stats.json`'s `detections.{hidden,flying,lead}` (parsed by `parseDetectionLevel()` — see [../decisions.md](../decisions.md) for the "Tower"-qualifier parsing rule). `uiController.handleAnalyzeWave` passes `level`/`detections` through on each tower object alongside `dps`/`pathCoverage`.
5. Damage cascades in wave order: each enemy's `damage` is added to the leftover (`carry`) from the previous enemy. If the total meets or exceeds that enemy's HP, it's killed; otherwise it survives and `carry` resets to 0. This approximates "the tower always focuses the frontmost enemy" without an actual per-enemy timeline. A stationary enemy (`speed <= 0`) is always killed if any tower has `pathCoverage > 0` (unbounded time available) but never produces `carry` itself — an unbounded carry would trivially kill everything behind it.

    **`carry` only propagates while a backlog exists.** After a kill, the new leftover (`damage - health`) is kept as `carry` for the next enemy *only if* `timeDeficit` (step 5b) is still `> 0` at that point — i.e. only if the next enemy is already queued up because this kill overran the spawn gap. If `timeDeficit` has drained back to `0` (kill finished well within the gap, nothing backed up behind it yet), the leftover is discarded instead of carried forward. **Why:** `damage` is computed from this enemy's own full path-crossing exposure window (`timeToCross`, step 3), which for a weak enemy killed almost instantly is enormously larger than its actual HP need — treating that entire unused window as free bonus damage for the next enemy (the pre-fix behavior) let a long run of trash enemies each donate a huge, physically meaningless surplus, snowballing until it could one-shot a high-HP enemy far down the wave with zero real damage dealt to it. Gating on `timeDeficit > 0` ties `carry` to an actual overlap in the field instead. See [../decisions.md](../decisions.md).

5b. **Spawn-timing backlog (`timeDeficit`)**: enemies are assumed to spawn `ENEMY_SPAWN_GAP_SECONDS` (fixed constant, `1`) apart, in a cumulative queue: after each kill, `timeDeficit = max(0, timeDeficit + killTime - 1)`, where `killTime` is the same value fed into `clearTime` below (floored at `0` — a deficit can never go negative, since an enemy can't be exposed longer than its own full path-crossing time). A kill slower than the gap grows the backlog; a run of kills faster than the gap drains it back down gradually rather than instantly. The current `timeDeficit` is subtracted from the *next* enemy's `timeToCross` (step 3) — killing something slowly eats into the next enemy's exposure window, since it's already further down the path by the time the tower is done — and, per step 5, gates whether `carry` propagates. Confirmed with the user: `timeDeficit` is **not** reset when an enemy survives or is `unavailable` — it's only updated on an actual kill, and otherwise passes through unchanged to whichever enemy comes next, crossing group boundaries. See [../decisions.md](../decisions.md).

## Defeat reason per group
`aggregateByGroup` reports `defeatReason` (`null` when the group is fully killed) for any group with survivors: `"No troop with <Type(s)> detection"` when zero placed towers are eligible at all for that enemy's active flags (`describeDefeatReason()` checks `canTowerHitEnemy` across all towers), `"Insufficient damage (<Type(s)>)"` when eligible towers exist but their combined DPS wasn't enough, or plain `"Insufficient damage"` for a non-Hidden/Flying/Lead enemy that's simply too tanky/fast.

## Clear time and risk badge
`computeEnemyDamage` also returns `rate` — the eligible towers' combined `dps * pathCoverage` — which is a per-second damage rate independent of the enemy's speed (speed only bounds total exposure time, not the rate). In the cascade, each killed enemy adds `max(0, health - carry) / rate` to a running `clearTime`, approximating "how long the towers spend killing enemies in this order" without a real per-enemy position timeline (same incremental-extension spirit as `carry`, not a new simulation). `evaluateWaveDamage` returns `{ groups, clearTime, allKilled, risk, waveTimerSeconds }`:
- `allKilled` is false if any group (or an `unavailable` entry) has survivors — `clearTime` is meaningless in that case.
- `waveTimerSeconds` comes from `getWaveTimerSeconds(wave.wave)`, a table of the official Hardcore Wave Timer (seconds before the next wave launches) duplicated from `pages/hardcore_damage_forecast.js` — see [../decisions.md](../decisions.md) for why it's duplicated rather than shared, and for the deliberate `Infinity` divergence on waves 43/45.
- `risk` is `"LOW"`/`"MEDIUM"`/`"HIGH"`: forced `"HIGH"` if `!allKilled`; otherwise `"LOW"` if `clearTime` fits within 80% of `waveTimerSeconds` (or the timer is `Infinity`), `"MEDIUM"` up to the timer, `"HIGH"` beyond it (`computeWaveRisk()`, see [../decisions.md](../decisions.md) for the 20% margin rationale).

`uiController.handleAnalyzeWave` passes all of this through to `sidebarView.renderAnalysis`, which renders the risk badge next to the wave status line and a `defeatReason` line under any row with survivors.

## Auto-refresh
No manual "Refresh" button — `uiController` subscribes to `placementModel.onChange` and re-runs analysis whenever the event is `"add"`, `"remove"`, `"clear"`, or an `"update"` whose changed keys include `level` (a level change goes through `updatePlacement(selected, { level, range, collision })` in `handleLevelChange`). Other updates (color, player, drag position) don't retrigger it. Both `handleAnalyzeWave`/`handleScanWaves` no-op until `analysisData` is loaded (first panel open), so this is safe to fire before the panel has ever been opened.

`uiController.analysisMode` (`"single"` or `"scan"`) tracks which view is current: `handleAnalyzeWave` sets it to `"single"`, `handleScanWaves` sets it to `"scan"`. `handlePlacementChangeForAnalysis` re-runs whichever one is current — so once the ❯❯❯❯ scan is open, placing/removing/re-leveling a tower re-scans (not a jump back to the single-wave view). Only an explicit pick in `waveSelect` (which calls `handleAnalyzeWave`) switches back to `"single"`.

## Wave scan (❯❯❯❯ button)
`#scanWaves` next to `waveSelect` triggers `uiController.handleScanWaves`: evaluates every wave in ascending order starting at Wave 1 (via the same `buildTowersForAnalysis`/`evaluateWaveDamage` used by `handleAnalyzeWave`), skips (doesn't display) any wave whose `risk` is `"LOW"`, keeps every `"MEDIUM"` wave, and stops at the first `"HIGH"` wave (included) — that's the earliest wave the current placement can't reliably handle. If no `HIGH` wave is found the scan just runs to the end of the wave list. `sidebarView.renderWaveScan` renders one `.wave-scan-block` per kept wave (each with its own risk badge and the same per-enemy-group rows as the single-wave view, via the shared `buildGroupRows` helper) instead of replacing `waveSelect`'s single-wave display.

## Relationship to the standalone hardcore pages
`pages/hardcore_damage_forecast.js` solves a related but separate problem (public pre-computed forecasts using the game's real Wave Timer, no placed-tower awareness) and does not share code with this feature — see [hardcore-pages.md](hardcore-pages.md). Don't assume a change to one applies to the other.
